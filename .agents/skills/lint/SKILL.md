# lint 技能：知識圖譜健康巡檢流水線

## 1. 核心目標
將軟體工程中的「靜態代碼分析 (Static Analysis)」引入知識管理。定期全面體檢，找出知識庫長期演進與多模型協作中產生的技術債（死鏈、孤島、認知衝突）。

## 2. 觸發條件
- 用戶輸入 `/lint` 或要求「檢查健康狀況」、「知識庫體檢」。

---

## 3. 四階段巡檢流水線 (SOP)

### 第 1 步：索引一致性檢查 (Index Check)
1. 讀取 `wiki/index.md` 的全部內容。
2. 掃描 `wiki/` 所有子目錄下的 `.md` 檔案（自動排除 index.md 與 log.md）。
3. 提取 index.md 中註冊的所有雙鏈連結 `[[頁面名稱]]`。
4. **比對**：找出「index 已註冊但實體檔案不存在」的**未同步索引**，以及「實體檔案存在但未在 index 註冊」的**黑戶頁面**。

### 第 2 步：雙向連結健康檢查 (Link Check)
1. 全局掃描所有 wiki 檔案內文，提取所有 `[[雙鏈]]` 格式。
2. 若該連結指向的頁面在本地完全不存在 → 標記為**死鏈 (Dead Link)**。
3. 統計每個頁面被其他檔案引用的次數。
4. 找出「從未被任何其他 wiki 檔案引用、且自己也沒有引用別人」的頁面 → 標記為**孤兒頁面 (Orphan Page)**。

### 第 3 步：認知衝突審查 (Conflict Check)
1. 全局檢索所有 wiki 檔案，找出內文包含 `## 知識衝突` 標題的頁面。
2. 提取每個衝突區塊的簡要描述與日期，列入技術債清單。

### 第 4 步：報告輸出規範
檢查完畢後，禁止靜默，必須在終端機打印出以下結構化報告：

```markdown
## 🩺 知識庫健康體檢報告 — YYYY-MM-DD

### ✅ 綠燈項
- [運行良好的項目，如：無孤兒頁面 / YAML 格式全數正確]

### ⚠️ 黃燈項
- **發現 N 個孤兒頁面**：[列表] - 建議手動添加關聯或分類
- **發現 N 個未同步索引**：[列表] - 檔案存在但未在 index.md 註冊

### ❌ 紅燈項
- **發現 N 個死鏈**：[來源頁面] → [[不存在的目標頁面]]
- **存在 N 個未解決的知識衝突**：[頁面名稱 — 衝突簡述]

### 🛠️ 下一步行動
1. 是否需要我自動修復未同步索引與簡單格式錯誤？
2. 是否需要針對特定的知識衝突進行多模型重新推演？
```

## 4. 硬約束
- **僅讀掃描**：生成報告前，禁止修改、刪除、重命名任何檔案
- **手動確認**：報告後等待用戶確認再執行修復
- **靜默日誌**：修復完成後，在 wiki/log.md 追加 `## [YYYY-MM-DD] lint | 修復了 N 個問題`
